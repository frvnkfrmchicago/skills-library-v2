# Stickerz Creation

Stickerz is a creation SYSTEM plus a family of feed FORMATS for big, custom,
cannabis, animated content. This skill documents the two headless producers an
agent (or a person, via the in-app maker) drives — the FILTER baker and the
SCENE-CARD producer — and the `create -> produce -> place` flow that lands a real,
verified artifact on a surface. No human in the loop: an agent invokes a runner
with flags and reads back the artifacts it writes (real files + JSON receipts).

Both CLIs copy the house contract from `scripts/generate-assets.ts` exactly:
flag-driven, single-JSON-stdout + a per-item receipt + a `_run-summary.json`, a
`--dry-run` plan, and **honest-empty over fake** — a real artifact or a documented
production-code-path, never a fabricated binary.

---

## The locked motion rule (read first)

Motion is **never generated by a paid AI-video API**. The generation router's
`ROLE_PREFERENCE.motion` resolves to a paid Hailuo video role; **neither CLI here
imports the router, a provider, a role, or a key.** Scene motion has exactly three
honest sources:

| Source | What it is | Producer |
| --- | --- | --- |
| **Vector** | a Lottie precomp scene (entrance-once + idle-loop) | `compileSceneToLottie` -> `validateLottie` -> `exportDotLottie` (`build-scene-cards.ts`) |
| **Raster** | a canvas-procedural scene (grinder / jar / smoke) | `AnimatedTemplateMaker` -> `encodeFramesToWebP` (the in-app / browser canvas forge) |
| **Video** | a looping muted **owned** clip | an EXISTING clip file the author supplies via `--clip` |

A GPU LUT (a filter) is image processing, not decorative animation — allowed. No
hand-coded CSS/SVG decorative animation. No mock/placeholder/seeded asset.

## Filters — `scripts/generate-filters.ts`

Bakes an authored **op-stack** JSON into a named, portable `.cube` 3D-LUT. The
op-stack is the PRO color grade the five-channel `AuthoredFilterConfig` can't carry
— per-channel + master tone curves, HSL band rotation, lift/gamma/gain, histogram
stretch, master saturation. The whole stack collapses, with **ZERO spend**, into a
`size^3` cube: `bakeToLut` (`lib/filters/op-stack.ts`) evaluates the op chain at
every grid point, and `writeCubeLut` (`lib/filters/lut.ts`) serializes the Adobe
`.cube`. The script re-implements none of that math — it sequences read -> validate
-> bake -> write -> receipt.

Because nothing contacts a provider, **`--dry-run` is a VALIDATE/CLAMP report**, not
a spend guard: it parses each input, runs the same clamp `bakeToLut` would, reports
what WOULD be written and any clamp correction, and writes no `.cube` bytes.

### The thumbnail seam (load-bearing)

The `app/api/media/filters` POST route **hard-requires a WebP thumbnail File**. A
`.cube` is not a thumbnail, and a real WebP needs a raster encoder bound to a canvas
(the browser forge / maker UI). So the headless baker does **not** fabricate a
thumbnail binary — each receipt carries a `thumbnail` plan naming the CPU reference
the forge samples (`applyLutToImageData` in `lib/filters/webgl-lut-pass.ts`, which
calls `sampleLutTrilinear` from `lib/filters/lut.ts`) over a swatch grid, plus the
`STICKER_SPEC`-class WebP target. The in-browser maker encodes + POSTs the real
WebP. Honest-empty over a fake thumbnail.

### Input shape

One or more `OpStack` JSON files (a file or a directory of them). A zero-op stack
bakes the **identity cube** (a valid no-op artifact), never a fabricated look.

```json
{
  "name": "Purple Punch",
  "size": 32,
  "ops": [
    { "type": "curves", "master": [{ "x": 0.2, "y": 0.16 }, { "x": 0.8, "y": 0.86 }] },
    { "type": "hsl", "bands": [{ "hue": 280, "range": 40, "hueShift": -8, "satScale": 1.2, "lightShift": 0.04 }] },
    { "type": "saturation", "amount": 1.12 }
  ]
}
```

### Invoke

```
bun scripts/generate-filters.ts [flags]
```

| Flag | Effect |
| --- | --- |
| `--in <path>` | op-stack JSON file OR a directory of them (default `filters/op-stacks`). |
| `--name <name>` | Pin the artifact name (single-file input). |
| `--size <17\|32\|64>` | Override the cube edge length. |
| `--out <dir>` | Artifact dir (default `public/filters`). |
| `--force` | Re-bake even if a `.cube` for this id exists. |
| `--dry-run` | Validate + clamp report; write NO `.cube`. |
| `--list` | Print discovered op-stack inputs as JSON and exit. |

```bash
# Validate + clamp report for every authored op-stack (writes nothing).
bun scripts/generate-filters.ts --dry-run

# Bake every op-stack in the input dir to .cube artifacts.
bun scripts/generate-filters.ts

# Bake one named filter at a 64-cube.
bun scripts/generate-filters.ts --in filters/op-stacks/purple-punch.json --size 64
```

Artifacts read back: `<out>/<id>.cube`, `<out>/<id>.receipt.json` (cube size +
points + bytes, the thumbnail plan, parse/clamp notes), `<out>/_run-summary.json`,
and the compact JSON on stdout.

## Scene-cards — `scripts/build-scene-cards.ts`

Produces the BIG contained scene-card in all three modes, from the named cannabis
scenes in `lib/stickerz/motion-registry.ts`. Every scene is first gated by
`validateScene` (`lib/stickerz/scene-model.ts`): an eased, **staggered** entrance,
**>= 2 depth planes**, a named **idle loop** that begins only after the entrance
settles (entrance-once-then-idle), and a **byte budget** checked on real produced
bytes (`checkByteBudget`).

Routing by content origin:

- **Vector (`lottie`)** — `compileSceneToLottie` builds a real Lottie document,
  `validateLottie` checks it, `exportDotLottie` packages the dotLottie (loop hint
  on, autoplay off so the runtime player gates entrance on viewport-enter). The
  receipt also carries `socialRasterPlan` (a transparent WebP + an opaque MP4) so
  the asset can leave the app — Lottie does not render on social.
- **Raster (`raster_webp`)** — validated, then a `raster` plan is emitted naming
  the canvas-procedural producer. The WebP is **not** baked headless (it needs a
  canvas-bound encoder); the in-app maker / browser forge bakes the real bytes.
- **Video** — supply an **existing owned clip** via `--clip` (with `--id` naming
  the scene it backs). The CLI verifies the real bytes and records them; an absent
  clip ships **honest-empty**. A clip is **never generated**.

### Invoke

```
bun scripts/build-scene-cards.ts [flags]
```

| Flag | Effect |
| --- | --- |
| `--id <sceneId>` | Only that registered scene (see `--list`). |
| `--format <lottie\|raster_webp>` | Only scenes of that content-origin format. |
| `--clip <path>` | An EXISTING owned clip to package as a VIDEO scene (needs `--id`). |
| `--out <dir>` | Artifact dir (default `public/stickerz/scenes`). |
| `--force` | Re-package even if an artifact exists. |
| `--dry-run` | Validate (scene + compiled Lottie) only; write nothing. |
| `--list` | Print registered scenes + the scene spec as JSON and exit. |

```bash
# See the scene catalogue + spec.
bun scripts/build-scene-cards.ts --list

# Validate every scene + its compiled Lottie (writes nothing).
bun scripts/build-scene-cards.ts --dry-run

# Package every vector scene to dotLottie.
bun scripts/build-scene-cards.ts --format lottie

# Back the video scene with an owned clip.
bun scripts/build-scene-cards.ts --id smoke-curl --clip public/videos/owned/smoke-loop.mp4
```

Artifacts read back: `<out>/<id>.lottie` (vector), `<out>/<id>.receipt.json`
(scene + Lottie validation, bytes, byte-budget verdict, the social/raster/video
plan), `<out>/_run-summary.json`, and the compact JSON on stdout.

## Create -> Produce -> Place

One core, two doors, one envelope, one placer. Every Stickerz item runs three
explicit stages:

1. **CREATE** — the per-type studio (or an agent authoring JSON) emits a DRAFT:
   an `OpStack` for a filter, a `SceneDefinition` (already in the motion registry)
   for a scene.
2. **PRODUCE** — one shared engine core lands real, verified bytes: `bakeToLut` ->
   `writeCubeLut` for a filter; `compileSceneToLottie` -> `exportDotLottie` (or the
   canvas forge / an owned clip) for a scene. The CLIs above are the headless door;
   `app/studio/*` + the standalone `maker-studio` are the self-serve door. Both
   consume the SAME engine, so a headless bake and an in-browser bake are
   byte-identical.
3. **PLACE** — a verified artifact is handed to one typed routing step that fans it
   out by surface (feed-block, story-tray, live-picker, sticker-library,
   scene-deck). A studio never writes straight to a surface.

### Wiring the place step (S14 registry — import by path)

The place step routes through lane S14's placement registry. Import it by path and
register a produced artifact ONLY after its real bytes are verified:

```ts
import { registerProducedArtifact } from '@/lib/media/placement-registry';
import type { ProducedArtifact } from '@/lib/media/produced-artifact';
```

After a CLI writes a real `.cube` (or a dotLottie / canvas WebP), the place step
builds the `ProducedArtifact` envelope from the receipt and calls
`registerProducedArtifact(artifact)`. The highest-leverage first wire: route the
baked filter `.cube` (plus F1–F30 + authored configs) through the registry into the
story-tray + camera, which makes "create a filter once, place it in story + live +
camera" real. Do not register an artifact whose bytes were not produced — the
registry is verify-real-bytes-before-register.

## NEVER

- NEVER call a paid AI-video API or the generation motion role for scene motion.
  Vector = Lottie/dotLottie, raster = canvas WebP, video = an owned clip.
- NEVER fabricate, seed, or placeholder an artifact. An absent owned clip, an
  un-encoded thumbnail, an un-baked raster WebP are honest-empty plans, not files.
- NEVER hand-code CSS/SVG decorative animation. (A GPU LUT is image processing —
  allowed.)
- NEVER register an artifact in the placement registry before its real bytes exist.
- NEVER spend during a filter bake — it is pure deterministic color math.
- NEVER edit the shared engines (`op-stack.ts`, `lut.ts`, `lottie-export.ts`,
  `scene-model.ts`, `motion-registry.ts`, the generation router) from a CLI — they
  are imported read-only.
